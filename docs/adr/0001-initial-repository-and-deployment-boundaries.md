# ADR 0001：初始仓库与部署边界

- 状态：Accepted
- 日期：2026-06-06
- 负责人：架构负责人
- 相关模块：M00-M11
- 相关任务：T003、T005、T007、T028、T040

## 背景

本项目是一个绿地企业级智能运维 Agent 平台。项目包含十二个逻辑模块，但如果初始阶段就创建十二个独立部署服务，会在模块行为尚未验证前引入过多发布、可观测、网络安全和运维成本。

执行链路的安全要求高于控制面。脚本和适配器执行不得与控制面进程或身份共享边界。

## 决策

使用一个单一仓库，并设置以下初始交付单元：

1. M01-M06 和 M09 后端行为采用模块化 Spring Boot 控制面。
2. M07 和已授权的 M08 执行器使用独立部署的受限执行 Worker。
3. M09 前端行为使用独立构建的操作台。
4. 设置版本化 Skill 区域和契约区域。
5. 部署、运维、文档和工具保存在同一仓库。

控制面内部仍需保持明确的逻辑模块边界。将模块拆分为新部署服务时，必须创建新的 ADR。

## 考虑过的备选方案

### 每个模块一个服务

V1.0 阶段拒绝采用，因为在扩展需求尚不明确时，会引入分布式事务、契约部署、网络安全和运维成本。

### 包含执行功能的单体应用

拒绝采用，因为控制面故障或失陷会与执行功能共享身份、文件系统、进程和网络边界。

### 多仓库

初始阶段拒绝采用，因为首期交付需要保持契约、安全规则和协同变更的一致性。

## 影响

正面影响：

- 初始交付更快，协同变更更简单。
- 执行隔离边界清晰。
- 契约和质量门禁可以统一管理。
- 后续可以根据实际证据拆分模块。

负面影响：

- 控制面内部必须严格遵守模块边界。
- 一次控制面发布可能包含多个模块变更。
- 随着代码增长，CI 需要支持选择性检查。

## 验证方式

- 使用架构测试验证控制面模块边界。
- 控制面无法直接执行脚本。
- Worker 使用独立身份和受限网络策略部署。
- 跨交付单元契约具有兼容性测试。

## 发布与回滚

在创建应用脚手架前，先建立仓库结构和契约。如果模块化控制面不再满足运维要求，必须创建新 ADR，明确服务边界、迁移顺序、契约和回滚策略。
