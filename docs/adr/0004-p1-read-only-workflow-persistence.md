# ADR 0004：P1 只读工作流持久化与恢复

- 状态：Accepted
- 日期：2026-06-07
- 负责人：Codex / project maintainers
- 相关模块：M05、M09
- 相关任务：P1 read-only workflow persistence

## 背景

P1 只读工作流已经具备路由、Worker 执行和语义事件输出，但状态、幂等和恢复仍停留在内存内，不满足执行事实源要求。

根据 `AGENTS.md`，关系型数据库必须作为执行事实源，且生产副作用控制链路需要持久化工作流、幂等和恢复语义。当前 P1 虽然只做只读诊断，但如果控制面在派发前后异常退出，系统无法从持久化状态恢复，也无法防止同一幂等请求被重复创建。

## 决策

P1 采用控制面内嵌的关系型工作流存储。

- 开发态使用文件型 `H2`。
- 访问方式采用 `R2DBC`，避免在 `WebFlux` 事件循环线程中执行阻塞 I/O。
- 迁移工件采用版本化 SQL 脚本管理，首个基线为 `sql/migrations/V001__workflow_schema.sql`。
- 只覆盖 `M05` 的只读工作流路径。
- 自动恢复只允许受控单次重放，并覆盖 `FAILED_RETRYABLE` 终态实例与当前 attempt 已过期的在途实例。
- 为了让控制面能够在恢复时重建同一份已授权请求，工作流实例会持久化原始只读命令信封。
- 为了判断在途实例是否已经失去活性，工作流存储会额外持久化 `workflow_attempt`，记录 attempt 编号、过期时间、失败分类和重放类型。
- 不在 P1 引入完整工作流引擎。

## 考虑过的备选方案

- 完整工作流引擎：对 P1 过重，会扩大范围并引入额外运维和建模成本。
- JDBC + `boundedElastic`：技术上可行，但会把阻塞 I/O 调度复杂度带入 P1。
- 单表幂等缓存：无法满足失败恢复与重放语义。

## 影响

正面影响：

- 满足执行事实源要求。
- 为幂等、失败恢复和事件重建提供落点。
- 后续切换正式关系库时可保留接口边界。

负面影响：

- 引入数据库初始化和数据迁移边界。
- `H2` 与未来正式数据库之间仍可能存在行为差异。

安全影响：

- 重放控制仍由控制面决定，Worker 不新增授权能力。
- 本决策不解决 `M07` 的生产传输认证和隔离问题。

## 验证方式

- H2 仓储集成测试
- 工作流状态迁移和恢复测试
- 幂等复用测试
- `backend` 全量 `verify`

## 发布与回滚

- 开发态使用 `var/workflow` 下的文件库。
- 回滚时可禁用新持久化装配并恢复到前一提交。
- 已生成的 H2 文件只作为开发事实源，不纳入构建产物或仓库提交。
