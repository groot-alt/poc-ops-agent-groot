# M09 当前工作流事件流自动恢复设计
- 日期：2026-06-07
- 相关任务：M09、M05、M11
- 目标阶段：P1 只读诊断 MVP
- 状态：待实施

## 1. 背景与目标

当前仓库已经具备以下 M09 基线能力：

- 控制面通过 `POST /internal/diagnostics/read-only/events` 输出强类型 SSE 语义事件；
- 只读工作流语义事件已持久化到关系型事实源；
- 前端最小只读操作台可以提交诊断请求并按事件类型渲染时间线。

当前缺口在于事件流可靠性不足：

- 前端连接断开后不能自动恢复；
- 前端只能重新发起整次诊断，无法从已落盘事件的最后进度继续订阅；
- 当前前端没有显式连接状态，用户无法区分“执行中但连接已断”与“工作流已结束”；
- 当前后端仅提供“启动工作流并返回完整事件流”的入口，没有“按既有工作流继续读取事件”的只读接口。

本次设计目标是在不扩大 P1 范围的前提下，完成“当前工作流内自动恢复”的闭环：

- 首次请求仍由现有启动入口创建只读工作流；
- SSE 断开后，前端自动重连；
- 重连时按 `workflowId + afterSequence` 从事实源恢复后续事件；
- 前端按强类型事件和有序序号去重，不从展示文本推断状态；
- 不引入生产写执行、历史工作流列表、浏览器端授权决策或执行中增量推送。

## 2. 设计范围

本次设计仅覆盖以下范围：

- 控制面新增只读事件恢复订阅接口；
- 工作流存储新增按 `workflowId` 和起始序号查询事件的能力；
- 操作台新增连接状态、自动重连、基于最近事件的续传与去重；
- 本地运行手册、验证说明和阶段证据更新。

本次设计明确不包含以下内容：

- 会话登录联调与 `/auth/session` 主入口切换；
- 历史工作流列表、查询页或手工选择旧工作流恢复；
- 执行中的增量事件推送；
- 任意脚本执行、生产写操作或新的审批交互；
- 多租户、外部客户接入或产品边界扩展。

## 3. 方案比较

### 方案 A：保留启动入口，新增恢复订阅入口

- 首次连接继续使用现有 `POST /internal/diagnostics/read-only/events`；
- 新增只读恢复接口，根据 `workflowId` 和 `afterSequence` 从持久化事件表回放未消费事件；
- 前端在首连成功后记录 `workflowId` 和最大 `sequence`，断线后自动切换到恢复接口。

优点：

- 复用现有 M05 事件持久化能力；
- 对现有启动接口影响最小；
- 职责边界清晰，启动与恢复分离；
- 符合 P1 “只补事件流可靠性、不改整体协议形态”的要求。

缺点：

- 需要新增一个查询接口与相应测试；
- 前端需要维护连接状态机。

### 方案 B：拆分为“创建工作流”和“订阅事件”两个新接口

- 先创建工作流并返回 `workflowId`；
- 再单独建立 SSE 订阅。

优点：

- 协议更标准；
- 后续扩展历史恢复与会话联调更自然。

缺点：

- 改动现有调用方式；
- 会扩大当前变更面；
- 不适合本轮以结果为导向的 P1 收敛目标。

### 方案 C：只做前端重试，复用原启动接口

- SSE 断开后，前端直接用原请求重新发起启动；
- 再依赖前端去重过滤重复事件。

优点：

- 实现成本最低。

缺点：

- 与当前幂等和进行中工作流语义冲突；
- 相同幂等键在工作流未结束时会命中“已在执行中”路径，无法稳定恢复；
- 实际上不能解决断点恢复问题。

推荐采用方案 A。

## 4. 总体设计

### 4.1 后端接口边界

保留现有首连入口：

- `POST /internal/diagnostics/read-only/events`

职责不变：

- 完成认证、授权、审计；
- 创建只读工作流；
- 返回当前工作流的完整语义事件序列。

新增恢复入口：

- `GET /internal/diagnostics/read-only/workflows/{workflowId}/events?afterSequence=<n>`

职责约束如下：

- 仅在服务端认证、授权和审计链路通过后访问；
- 仅按持久化事实源读取指定工作流中 `sequence > afterSequence` 的事件；
- 不创建新工作流；
- 不重新路由 Skill；
- 不重新提交 Worker；
- 不通过展示文本推断状态。

### 4.2 存储与查询

复用 `ReadOnlyWorkflowStore` 及其 R2DBC 实现，新增事件查询能力：

- 根据 `workflowId` 读取工作流视图；
- 按 `workflowId` 和 `afterSequence` 顺序查询事件；
- 查询结果按 `sequence asc` 返回。

服务端需要保证：

- 只能读取存在的工作流；
- 恢复查询出的事件仍是 `contractVersion=1.0` 的强类型语义事件；
- 不泄露未受控的内部状态。

### 4.3 前端状态机

前端新增当前工作流事件流状态：

- `idle`：未发起诊断；
- `connecting`：首次建立事件流；
- `streaming`：连接正常；
- `reconnecting`：连接中断后重试恢复；
- `completed`：收到终态事件且流结束；
- `failed`：无法恢复或请求失败。

前端持久化于内存中的最小恢复上下文：

- `workflowId`
- `lastSequence`
- `receivedEventIds`
- 当前连接阶段

其中：

- `workflowId` 来自首个事件；
- `lastSequence` 为当前已渲染最大 `sequence`；
- `receivedEventIds` 仅用于当前页面会话内去重；
- 不引入浏览器本地持久化，不跨刷新恢复。

### 4.4 自动恢复与去重

前端恢复策略如下：

1. 首次请求成功后，开始消费事件；
2. 每接收一条事件，更新 `workflowId`、`lastSequence` 和去重集合；
3. 如果流在未进入终态前异常关闭，则进入 `reconnecting`；
4. 使用恢复接口发起续传，请求参数为最新 `workflowId` 和 `lastSequence`；
5. 对恢复返回的事件继续按 `eventId` 去重，并以 `sequence` 递增追加；
6. 如果恢复成功并读到终态事件，则状态转为 `completed`；
7. 如果恢复接口返回错误，状态转为 `failed` 并展示明确错误。

本次不引入无限重试。建议使用有限次自动重连，例如 3 次，带短退避，避免前端在不可恢复错误下持续占用连接。

### 4.5 终态判定

前端终态仅由强类型事件决定：

- `WORKFLOW_COMPLETED`
- `WORKFLOW_FAILED`

出现上述任一事件后：

- 更新界面为终态；
- 停止自动重连；
- 即使后续连接关闭，也不再尝试恢复。

## 5. 失败处理

需要覆盖以下场景：

- 首次启动请求返回 `401`；
- 首次启动请求返回 `403`；
- 首次启动请求返回其他非 `2xx`；
- SSE 在终态前网络断开；
- 恢复接口返回工作流不存在；
- 恢复接口返回无权限；
- 恢复接口返回空事件集但工作流已终态；
- 恢复接口返回重复事件。

处理原则如下：

- `401` 与 `403` 前端要明确区分，不混淆成“网络错误”；
- 非授权类错误展示结构化错误信息；
- 重复事件必须被去重，不产生重复时间线项；
- 空恢复结果允许存在，但不能错误地把未终态工作流标记为完成；
- 前端不得自行推断安全状态，只显示服务端返回的强类型事件与请求结果。

## 8. 风险与约束

- 当前后端事件流仍是“工作流完成后回放完整事件序列”，恢复接口只能恢复已落盘事件，不能把体验描述成真正的执行中流式推送；
- 若未来切换为执行中增量 SSE，本设计的前端状态机可复用，但后端实现需要再扩展；
- 若当前工作流在服务端仍未终态而连接断开，恢复接口可能暂时拿不到新增事件，本轮允许前端有限次重试后失败提示；
- 工作区已有进行中的 M05 改动，本次实现必须避免覆盖无关文件内容。

## 9. 验收标准

满足以下条件后，本次 M09 事件流恢复切片才算完成：

- 首次只读诊断仍可按原入口正常发起；
- 前端在终态前遇到连接中断时会自动进入恢复流程；
- 恢复请求按 `workflowId + afterSequence` 从持久化事件继续读取；
- 前端不会渲染重复事件；
- 前端显式展示连接中、恢复中、完成和失败状态；
- 未认证请求返回 `401`，无权限请求返回 `403`；
- `npm run build` 通过；
- 相关后端测试通过；
- 本地运行手册与阶段证据更新为中文。
