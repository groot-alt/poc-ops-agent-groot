# M09 基于原型的操作台首轮重写设计

- 日期：2026-06-13
- 相关模块：M01、M03、M04、M09、M11
- 目标阶段：P1 只读诊断 MVP
- 状态：已批准，待实施

## 1. 背景与目标

当前 `frontend/operator-console` 已包含最小 React 操作台和正在开发的 SQL 工作台页面，但现有页面结构与已确认的产品原型不一致。本次重写以 `figma-prototype/ops-agent-aia-prototype.html` 及其页面截图为视觉事实源，删除现有页面实现并重新构建操作台。

首轮只实现以下四个页面：

1. 登录页。
2. Agent 工作台。
3. Skill 注册中心。
4. SQL 工作台。

本次设计目标：

- 忠实还原原型的桌面端视觉语言和首屏布局。
- 建立可复用、可测试的轻量页面组件体系。
- 优先接入仓库已有真实接口，不使用运行时 Mock 数据伪装后端能力。
- 对缺失接口、未通过评测门禁或超出 P1 范围的能力明确禁用并说明原因。
- 保持 M09 只展示服务端返回的授权、安全和执行状态，不在浏览器中做授权决策。

## 2. 范围与非目标

### 2.1 首轮范围

- 重写登录页、Agent 工作台、Skill 注册中心和 SQL 工作台。
- 建立四页共享的应用外壳、顶部导航、设计令牌、基础组件和反馈组件。
- 登录页对接现有浏览器会话、登录和退出接口。
- SQL 工作台对接现有连接目录和 SQL 校验接口。
- Agent 工作台和 Skill 注册中心优先接入已有真实接口。
- 对真实接口返回的数据执行运行时 Schema 校验。
- 建立组件测试、页面流程测试和桌面端视觉验收基线。
- 同步修订前端工具链 ADR，将开发语言从 TypeScript 严格模式调整为 JavaScript、JSX、JSDoc 和 `checkJs`。

### 2.2 明确不包含

- 平台总览、RAG 问答、诊断工作台、工作流事件和审计记录页面。
- 手机端和平板端完整响应式适配。
- 生产写执行、任意脚本执行或审批绕过。
- 在浏览器中判断用户是否有权执行某项操作。
- 使用 Mock 数据伪装尚不存在的生产能力。
- 自动执行 AI 生成内容。
- Agent 模型内部推理过程展示。
- Skill 安装、升级或卸载等未由服务端提供并授权的操作。

未实现页面首轮不提供虚假可用入口。后续扩展时复用本次建立的应用外壳与组件体系。

## 3. 方案比较与选择

### 3.1 方案 A：按功能切片逐页实现

先建立设计令牌、基础组件、应用外壳和路由，再按登录页、Agent 工作台、Skill 注册中心、SQL 工作台逐页实现、接入和验收。

优点：

- 每个页面形成独立、可测试的工作切片。
- 可以逐页进行原型视觉对比和真实接口验收。
- 容易控制重写风险，避免一次性替换后无法定位问题。

缺点：

- 共享组件会随页面推进逐步完善，需要保持边界克制。

### 3.2 方案 B：先完成四页视觉，再统一接入接口

优点：

- 能较快看到完整页面外观。

缺点：

- 真实数据状态、错误状态和权限拒绝接入时可能迫使页面大范围返工。
- 容易形成以静态样例数据为中心的组件接口。

### 3.3 方案 C：直接把原型 HTML 改造成 React

优点：

- 初期复制速度快。

缺点：

- 原型包含大量页面专用 CSS、内联样式和演示数据。
- 难以形成清晰组件边界和真实数据流。
- 长期维护、测试和可访问性成本高。

采用方案 A。

## 4. 技术基线

### 4.1 核心技术

- React 19：页面和交互组件。
- JavaScript 与 JSX：业务和组件实现语言。
- JSDoc 与 `checkJs`：为 JavaScript 提供静态类型检查。
- Vite 6：开发和生产构建，延续已接受的前端工具链决策。
- React Router：四页路由、受保护页面跳转和登录后导航。
- TanStack Query：管理会话、Skill 列表、SQL 连接等服务端状态。
- Zod：校验 API、SSE 和其他外部边界数据。
- Lucide React：提供统一图标，避免手写 SVG 和文本符号模拟图标。
- Monaco Editor：为 SQL 工作台提供 SQL 语法高亮、行号和快捷键。
- CSS Modules 与 CSS Variables：组件样式隔离和设计令牌管理。
- Vitest、React Testing Library：组件和页面状态测试。
- Mock Service Worker：仅在自动化测试中模拟服务端响应。
- Playwright：浏览器流程和桌面端页面验收。

### 4.2 JavaScript 变更约束

当前全局工程契约和 ADR 0003 要求 TypeScript 严格模式。本次采用 JavaScript 是明确的技术决策变化，实施前必须：

- 修订 `docs/adr/0003-operator-console-toolchain.md`，记录调整理由、影响和回滚方式。
- 同步更新涉及前端技术基线的 Markdown 事实源。
- 在 JavaScript 文件中启用 `checkJs`。
- 对公共函数、API 数据、组件属性和复杂状态使用 JSDoc 声明。
- 使用 Zod 对所有外部数据做运行时校验。
- 禁止通过关闭检查或广泛使用无约束对象绕过类型与契约要求。

### 4.3 第三方依赖评估

首轮计划引入的依赖均应使用宽松开源许可证，并在实施时通过锁文件和许可证检查确认直接依赖与传递依赖。

| 依赖 | 用途 | 预期许可证 | 安全与维护影响 |
|---|---|---|---|
| React Router | 页面路由 | MIT | 路由不能承载授权决策 |
| TanStack Query | 服务端状态 | MIT | 缓存不能作为授权或执行事实源 |
| Zod | 运行时数据校验 | MIT | 只保护前端边界，不替代服务端校验 |
| Lucide React | 图标 | ISC | 只用于展示 |
| Monaco Editor | SQL 编辑器 | MIT | 增加构建体积，需要按页面懒加载 |
| Vitest | 单元与组件测试 | MIT | 仅开发与 CI 使用 |
| React Testing Library | 组件交互测试 | MIT | 仅开发与 CI 使用 |
| Mock Service Worker | 测试响应模拟 | MIT | 禁止用于生产运行数据 |
| Playwright | 端到端测试 | Apache-2.0 | 仅开发与 CI 使用 |

不引入 Ant Design、MUI、Element Plus 等完整 UI 组件库。本项目原型具有明确的定制视觉语言，完整组件库会带来大量覆盖样式和不必要依赖。按钮、卡片、徽标、导航和反馈状态使用项目自有轻量组件。

## 5. 前端架构

### 5.1 目录结构

```text
frontend/operator-console/src/
|-- app/
|   |-- App.jsx
|   |-- router.jsx
|   `-- providers.jsx
|-- api/
|   |-- client.js
|   |-- auth-api.js
|   |-- agent-api.js
|   |-- skill-api.js
|   `-- sql-api.js
|-- schemas/
|   |-- auth-schemas.js
|   |-- agent-schemas.js
|   |-- skill-schemas.js
|   `-- sql-schemas.js
|-- components/
|   |-- layout/
|   |-- primitives/
|   |-- feedback/
|   `-- data-display/
|-- features/
|   |-- auth/
|   |-- agent-workspace/
|   |-- skill-registry/
|   `-- sql-workbench/
|-- styles/
|   |-- tokens.css
|   |-- reset.css
|   `-- global.css
`-- main.jsx
```

职责边界：

- `app` 只负责应用启动、路由和全局 Provider。
- `api` 只负责 HTTP、SSE、错误归一化和请求封装。
- `schemas` 只负责外部数据运行时校验。
- `components` 只包含跨页面复用且不依赖具体业务接口的组件。
- `features` 按页面业务功能组织页面、查询 hooks、局部组件和状态。
- 页面组件不得直接调用 `fetch`。

### 5.2 路由

首轮路由：

- `/login`
- `/agent`
- `/skills`
- `/sql`
- `/`：读取会话后跳转至 `/agent` 或 `/login`

路由只负责导航和页面可见性。是否允许执行操作仍由服务端策略决定。未认证用户进入受保护页面时跳转登录页；服务端返回 `403` 时页面展示拒绝结果，不将其伪装成路由不存在。

### 5.3 现有代码处理

实施阶段删除现有页面实现和旧页面样式，包括当前单文件页面结构。以下内容可以保留或迁移：

- `package.json`、Vite 配置和构建入口等工具链配置。
- 已验证可用的真实接口路径和请求语义。
- 后端版本化契约及其字段定义。

现有 TypeScript 页面和 API 文件不得直接作为新页面结构继续扩展。可复用的接口逻辑必须迁移到新的 JavaScript API 模块，并通过 Zod 校验。

## 7. 页面设计

### 7.1 登录页

职责：

- 忠实还原原型登录视觉。
- 读取 `/auth/session`。
- 使用现有 `/auth/login` 登录入口。
- 使用现有 `/logout` 退出入口。
- 展示会话读取中、未认证、认证失败和已认证跳转状态。

登录成功后跳转 `/agent`。登录页不收集或持久化服务端未要求的凭据，也不在浏览器中推断角色权限。

### 7.2 Agent 工作台

职责：

- 展示任务交流区、可审计计划摘要和任务状态。
- 仅展示服务端返回的计划摘要和强类型状态。
- 不展示模型内部推理过程。
- 不允许模型输出直接授予权限或触发越界执行。

若仓库现有接口不足以支持原型中的任务发送或执行：

- 对应按钮保持禁用。
- 使用 `DisabledFeature` 明确显示缺失接口或门禁原因。
- 不使用 Mock 响应模拟成功。

### 7.3 Skill 注册中心

职责：

- 展示真实 Skill 列表及可用字段，包括 Skill ID、分类、风险、版本、Owner、执行器和发布状态。
- 支持基于现有只读接口的搜索和筛选。
- 默认突出 P1 只读 Skill。

安装、升级、卸载和发布等操作仅在服务端存在明确接口、授权和审计链路时开放。首轮缺失操作保持禁用，不模拟成功。

### 7.4 SQL 工作台

职责：

- 对接 `GET /internal/sql-workbench/connections`。
- 对接 `POST /internal/sql-workbench/queries/validate`。
- 仅展示服务端返回的开发和测试连接。
- 支持只读执行校验、静态校验和 DML 预检。
- 使用 Monaco Editor 提供 SQL 高亮、行号和快捷键。
- 展示连接、Schema、环境、平台、校验报告、风险、拒绝原因和未验证项。

限制：

- 浏览器不直接访问 Worker 或数据库。
- P1 不提供 DML 执行、交互事务、提交或回滚。
- 生产连接不可见、不可调用。
- AI SQL 助手保持禁用，并说明尚未通过模型评测门禁。

## 8. 数据流与错误处理

### 8.1 数据流

统一数据流：

```text
页面交互
  -> feature query 或 mutation
  -> API 模块
  -> 控制面接口
  -> Zod Schema 校验响应
  -> TanStack Query 服务端状态
  -> 页面渲染
```

前端不得把 TanStack Query 缓存、浏览器存储或展示文本作为授权或执行事实源。

### 8.2 错误分类

- `401`：会话失效或未认证，跳转登录页并保留可理解提示。
- `403`：展示服务端拒绝原因，不改写为普通网络错误。
- 其他非 `2xx`：展示稳定的结构化请求错误。
- 网络失败：展示可重试状态，不模拟成功。
- Zod 校验失败：展示契约不兼容错误并阻止错误数据进入页面。
- 空数据：展示真实空状态。
- 缺失能力：展示禁用状态和原因。

外部数据、用户输入、模型输出和 SQL 内容一律视为不可信数据。Zod 校验只保护前端边界，不能替代服务端认证、授权、审计和 SQL 安全校验。

## 9. 测试与验收

### 9.1 自动化测试

组件和页面测试必须覆盖：

- 登录会话加载、未认证、认证失败和成功跳转。
- 受保护路由重定向。
- Agent 缺失接口能力的禁用状态。
- Skill 列表成功、空数据、`403` 和契约校验失败。
- SQL 连接加载、校验请求、风险报告、拒绝结果和未验证项。
- `401`、`403`、网络失败、空数据和 Zod 校验失败。
- 导航、按钮、Tab 和关键输入交互。

Mock Service Worker 仅用于测试上述状态，不进入生产构建和产品运行路径。

### 9.2 浏览器与视觉验收

使用 Playwright 和人工视觉比对验证：

- 登录页、Agent 工作台、Skill 注册中心和 SQL 工作台。
- `1280px`、`1440px` 和 `1920px` 桌面宽度。
- 原型中的导航、页面标题、卡片层级、间距、圆角、边框、颜色和字体层级。
- SQL 编辑器、结果区和禁用 AI 助手布局。
- 无破损布局、裁切、错误间距和不可访问关键操作。

视觉比对时使用相同视口，将原型截图与实现截图并排检查。截图本身不作为唯一验收证据，关键交互必须实际执行。

### 9.3 工程门禁

声明完成前必须通过：

- JavaScript `checkJs`。
- ESLint。
- Vitest 和 React Testing Library 测试。
- Playwright 端到端测试。
- 生产构建。
- 依赖许可证与已知安全问题检查。
- 浏览器端授权边界和 P1 禁止项检查。

## 10. 风险与缓解

### 10.1 JavaScript 降低静态类型强度

风险：JavaScript 相比 TypeScript 更容易让不完整类型进入实现。

缓解：

- 强制启用 JSDoc 与 `checkJs`。
- 所有外部数据使用 Zod。
- API 与复杂状态集中定义，不允许无约束对象跨功能流动。
- 修订 ADR 并明确回滚方案。

### 10.2 原型与真实接口能力不一致

风险：原型包含尚不存在或超出 P1 的能力。

缓解：

- 真实接口优先。
- 缺失能力明确禁用。
- 不使用运行时 Mock 数据伪装成功。
- 以服务端契约和项目事实源确定可用行为。

### 10.3 Monaco Editor 增加构建体积

风险：SQL 编辑器会显著增加前端资源体积。

缓解：

- 仅在 SQL 路由中懒加载。
- 只加载 SQL 所需语言能力。
- 在构建门禁中记录和检查产物体积。

### 10.4 工作区存在并行改动

风险：当前工作区已有 SQL 工作台和后端未提交改动。

缓解：

- 实施时先确认并行改动归属。
- 不回退或覆盖用户已有改动。
- 页面重写与后端接口变更分开提交和评审。

## 11. 发布与回滚

- 页面重写按登录页、Agent 工作台、Skill 注册中心和 SQL 工作台分切片提交。
- 每个切片必须可以独立构建和测试。
- 静态前端制品部署前完成四页浏览器验收。
- 回滚通过部署上一已验证前端静态制品完成。
- 若 JavaScript 工具链调整无法满足质量门禁，回滚 ADR 变更并恢复 TypeScript 严格模式。

## 12. 验收标准

满足以下全部条件后，首轮重写才算完成：

- 现有页面实现和旧页面样式已被新架构替换。
- 登录页、Agent 工作台、Skill 注册中心和 SQL 工作台均可通过路由访问。
- 四页桌面端视觉与原型目标一致，并通过三个目标宽度验收。
- 登录和 SQL 工作台接入现有真实接口。
- Agent 工作台和 Skill 注册中心优先使用真实接口，缺失能力明确禁用。
- 运行时产品不使用 Mock 数据伪装后端能力。
- 所有外部数据经过 Zod 校验。
- 前端不做授权决策，不从展示文本推断安全状态。
- P1 不存在生产写入、任意脚本执行或审批绕过入口。
- JavaScript `checkJs`、ESLint、组件测试、端到端测试和生产构建通过。
- ADR 0003 和相关 Markdown 事实源已同步更新。
- 依赖许可证、安全影响、发布和回滚方式已记录。
