# AGENTS.md

## 1. 文档目的

本仓库用于实现“企业级智能运维 Agent 开放底座”。本文件是所有人工贡献者和 AI 编码 Agent 必须遵守的全局工程契约。

V1.0 的交付策略：

1. 首先交付只读诊断 MVP。
2. 然后增加受控、低风险、可回滚的变更能力。
3. 最后完成生产加固、安全验证和运维移交。

当局部模块规则与本文件冲突时，采用更严格的安全规则。架构或安全例外必须编写 ADR，并经过明确评审。

## 2. 事实源优先级

按以下优先级使用项目资料：

1. `AGENTS.md`：全局工程与交付规范。
2. `docs/architecture/module-map.md`：模块归属和依赖边界。
3. `docs/adr/` 中已接受的 ADR：技术决策。
4. `docs/planning/project-plan.md`：阶段、里程碑和当前启动计划。
5. `docs/planning/design-traceability.md`：设计范围与规划追溯。
6. `backend/contracts/` 中的版本化契约：API、事件、Skill 和工作流 Schema。
7. 各模块 README：模块职责和开发说明。

仓库中的 DOCX/XLSX 规划文件属于参考产物。当其内容转化为实际仓库工作时，必须在同一次变更中同步更新 Markdown 事实源。

## 3. 总体架构形态

V1.0 使用单一仓库，包含四个主要交付单元：

本系统定位为公司内部自研自用、单组织部署。设计中没有租户或多租户概念；未经新的设计决策和用户确认，禁止引入租户隔离、租户标识、租户路由、租户计费或租户管理能力。

- `backend/control-plane`：模块化 Spring Boot 控制面。
- `backend/execution-worker`：独立部署的受限执行 Worker。
- `frontend/operator-console`：操作、审批和可观测界面。
- `backend/skills`：版本化 Skill 定义、测试和目标系统适配器。

配套区域：

- `backend/contracts`：跨模块版本化 Schema 和兼容性测试。
- `backend/deploy`：环境、部署定义和运维配置。
- `docs`：架构、ADR、规划、规范和运行手册。
- `tools`：仓库内开发与 CI 工具。

不要仅因为存在一个模块，就创建一个独立部署服务。只有当隔离、扩展、发布归属或可用性需求足以证明额外运维成本合理时，才可以拆分服务，并且必须记录 ADR。

## 4. 模块边界

标准模块编号：

- `M00` 项目治理与工程基础
- `M01` 接入网关与身份认证
- `M02` 策略授权与审计
- `M03` Skill 契约与注册中心
- `M04` Agent 路由与模型交互
- `M05` 工作流、审批与状态恢复
- `M06` DAG 编排与制品黑板
- `M07` 执行器与安全隔离
- `M08` 运维 Skill 与目标系统集成
- `M09` 操作台与语义事件流
- `M10` 可观测性与平台运维
- `M11` 测试评测、安全验证与发布

规则：

- 跨模块调用必须使用明确接口或版本化契约。
- 禁止通过访问其他模块的数据表来绕过模块 API。
- 控制面不得直接执行脚本，也不得直接使用目标系统长期凭据。控制面只能向执行 Worker 提交已授权工作。
- 执行 Worker 不得自行进行授权决策，只能接受已授权、带版本的执行请求。
- 操作台不得绕过策略、审批、幂等、审计或隔离控制。
- `M11` 的测试与发布门禁必须能够独立验证。

模块职责和依赖详情见 `docs/architecture/module-map.md`。

## 5. 不可妥协的安全规则

- 将模型输出、用户输入、Skill 输入、脚本输出和制品全部视为不可信数据。
- 模型可以提出意图或计划，但永远不能授予权限。
- 服务端策略是唯一授权决策点。
- 人格、Prompt、客户端输入和直接 API 调用可以提高限制，但绝不能降低组织安全基线。
- 生产副作用操作必须具备持久化工作流、幂等控制、审计事件，以及明确的补偿或人工接管路径。
- 审批必须绑定准确的 Skill 版本、目标、参数哈希、策略版本、审批人范围和过期时间。关键内容变化后审批自动失效。
- Redis 和 ChatMemory 不得作为执行事实源。
- 禁止在源码、日志、Prompt、制品、环境变量示例或测试数据中存储密钥。
- Worker 必须使用最小权限身份、受限工作区、受控网络出口和短期凭据。
- 高风险或不可信执行必须进入强隔离环境。Job Object 和 WDAC 仅属于纵深防御，不是完整隔离边界。
- 禁止暴露模型内部推理过程。只能输出可审计的计划摘要和执行状态事件。

如果任务会违反或削弱以上任何规则，必须停止实现并请求安全或架构决策。

## 6. 技术基线

初始技术基线：

- 后端：Java 21、Spring Boot 3.2+、Spring AI 1.0+、WebFlux。
- 后端构建：Maven 多模块构建。后端开发开始前必须添加并提交 Maven Wrapper。
- 前端：JavaScript/JSX、React、JSDoc 和 TypeScript `checkJs` 技术栈的单页应用。应用脚手架创建前必须通过 ADR 确定具体工具。
- 数据：关系型数据库作为执行事实源；Redis 用于缓存、锁、限流和短期会话；受控对象存储用于大型制品。
- 契约：按场景使用 JSON Schema 和 OpenAPI。
- 可观测性：结构化日志、指标、链路追踪和集中审计事件。

工作流引擎、策略引擎、对象存储实现和部署平台仍属于待决 ADR。禁止在未经评审的情况下把某个选择嵌入核心代码。

## 8. 契约规则

- 实现依赖合入前，必须先对契约进行版本化和评审。
- 优先采用向后兼容的增量变更。
- 破坏性变更必须创建新契约版本，并提供迁移方案、兼容性测试以及发布和回滚方案。
- 每个命令必须包含操作人身份上下文、目标环境、Skill 版本、幂等键、必要的策略或审批引用，以及链路追踪上下文。
- 每个语义事件必须包含 `eventId`、`workflowId`、`sequence`、`timestamp`、`type` 和类型明确的 `payload`。
- 禁止在完整执行链路中使用无约束的 `Map<String, Object>`。动态载荷必须受 Schema 约束，并封装在强类型信封中。

## 9. 编码规范

通用规则：

- 代码、标识符、文件名、API 字段和提交标题使用英文。
- 仓库内所有新增、更新和产出的文档必须使用中文；仅当引用外部原文、协议标准字段或用户明确要求时，才可保留必要英文。
- 优先编写清晰、明确的代码，避免炫技式抽象。
- 变更范围必须聚焦，禁止把无关重构混入功能开发。
- 新增依赖前，必须说明用途、许可证与安全影响，以及现有技术栈为何不足。

Java 规则：

- 按模块划分包结构，模块内部实现默认不对外公开。
- 命令、决策和事件优先使用不可变值对象。
- 在所有信任边界进行校验，不依赖前端或模型校验。
- 阻塞 I/O 不得运行在 WebFlux 事件循环线程中。
- 未经 ADR 批准，不得在生产代码中使用 Java 预览特性。
- 基础设施错误必须转换为稳定的领域或应用错误契约。

JavaScript/JSX 规则：

- 使用 JSDoc 声明数据与组件契约，并开启严格 `checkJs` 静态检查。
- API、SSE 和其他外部边界数据必须使用 Zod 解析和校验，不得仅依赖 JSDoc 声明。
- 浏览器中不得包含授权逻辑。
- 前端必须渲染强类型语义事件，不得从展示文本推断安全状态。

Skill 定义规则：

- Skill 必须包含 Owner、版本、分类、风险、执行器、Schema、权限、超时、重试策略、输出契约和测试用例。
- 生产 Skill 必须经过签名、评审、测试，并支持回滚。
- 只有只读、幂等或明确可安全重试的操作才允许自动重试。

## 10. 测试与质量门禁

每项变更都必须包含与风险相匹配的测试。

必须覆盖的测试层：

- 领域和校验逻辑的单元测试。
- API、事件、Skill Schema 和 Worker 命令的契约测试。
- 持久化和外部适配器的集成测试。
- 授权拒绝和边界强制执行的安全测试。
- 副作用操作的工作流恢复与幂等测试。
- 路由、计划生成和安全拒绝行为的模型评测。

以下情况阻止发布：

- 存在未关闭的严重或高危安全问题。
- 存在绕过策略、审批、审计、幂等或隔离的路径。
- 副作用操作没有恢复语义。
- 契约不兼容且没有迁移方案。
- 关键工作流缺少可观测性。
- 必要测试、评测阈值、恢复演练或回滚验证失败。

## 11. 可观测性与审计

- 传递 trace、workflow、task、operator、Skill 和 version 标识。
- 使用结构化日志，禁止通过解析展示文本推断运行状态。
- 审计授权决策、审批、执行请求、结果、补偿、人工接管、配置变更和发布。
- 日志、事件、Prompt 和制品落盘前必须脱敏。
- 每条关键路径在实现时就必须定义指标和告警，不得在开发结束后补做。

## 12. 开发工作流

编码前：

1. 明确任务 ID、模块 ID、验收标准、依赖和风险。
2. 阅读相关模块 README、契约和 ADR。
3. 跨模块行为必须先创建或更新契约。
4. 确认安全和可观测性要求。

编码中：

1. 实现必须遵守模块边界。
2. 行为实现与测试必须同时提交。
3. 关键行为必须添加指标、日志、追踪和审计事件。
4. 决策变化时必须更新文档和 ADR。

声明完成前：

1. 运行相关格式化、静态检查、测试和契约检查。
2. 验证授权拒绝、失败和恢复路径。
3. 确认没有添加密钥或敏感制品。
4. 更新任务状态和验证证据。
5. 记录部署和回滚影响。

## 13. Git 与评审规则

- 分支名使用 `codex/`、`feature/`、`fix/`、`docs/` 或 `chore/` 前缀。
- Commit 标题使用祈使语气英文，并描述一个完整变更。
- PR 描述必须引用任务 ID 和模块 ID。
- PR 必须说明范围、风险、测试、契约变更、发布和回滚方式。
- 安全关键模块需要安全评审：`M02`、`M05`、`M07`，以及 `M08` 中的生产变更。
- 禁止提交构建产物、密钥、本地运行状态或生产数据。

## 14. 完成定义

只有满足以下全部条件，任务才算完成：

- 验收标准已被证明满足。
- 代码和契约已通过评审。
- 必要测试和安全检查通过。
- 适用时已验证失败、恢复和回滚行为。
- 已覆盖指标、日志、追踪和审计。
- 文档和运行手册已更新。
- 不存在未解决的阻塞依赖或隐藏人工步骤。

## 15. 当前项目优先级

当前优先级是阶段 P1：只读诊断 MVP。

P1 里程碑验收之前：

- 禁止实现生产写执行。
- 禁止启用任意脚本执行。
- 禁止添加绕过策略或审计的路径。
- 优先建设契约、身份、策略、审计、只读 Skill 路由、语义事件、评测和 CI 基础。

## 16. 设计范围约束

- 架构规划必须严格以 V8.1 设计蓝图为范围依据。
- 不得因为“常见企业平台做法”擅自增加多租户、计费、商业化、外部客户接入等能力。
- 为了落地安全、可靠性或工程交付而补充的机制，必须能够追溯到设计目标，并在 ADR 中说明原因。
- 如果设计没有明确表达且补充内容会改变产品边界，必须先向用户确认，不得直接写入全局规范或开发任务。
